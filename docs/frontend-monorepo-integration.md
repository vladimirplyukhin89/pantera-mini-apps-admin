# Подключение сайта к Strapi и вынос админки из монорепозитория

Гайд для схемы: **фронтенд** (сайт) живёт в **монорепозитории**, **Strapi** — в этом репозитории `pantera-mini-apps-admin`; из монорепо папку с CMS нужно убрать и переключить сайт на внешний API.

## Целевая схема

| Компонент | Репозиторий | Деплой | Публичный URL (пример) |
|-----------|-------------|--------|-------------------------|
| Сайт (Astro/React и т.д.) | монорепо `pantera-mini-apps` | Beget / CI | `https://pantera-boxing.ru` |
| Strapi (админка + API) | `pantera-mini-apps-admin` | VPS | `https://admin.pantera-boxing.ru` |

Связь односторонняя: сайт **только читает** API по HTTPS (URL + при необходимости read-only API token). В монорепозитории **нет** кода Strapi.

## Что должно быть готово до переключения фронта

1. Strapi доступен по HTTPS: `https://admin.pantera-boxing.ru/admin` открывается, API отвечает (например `GET /api/...`).
2. В Strapi задан **`PUBLIC_URL=https://admin.pantera-boxing.ru`** (или актуальный домен) — см. `config/server.ts`.
3. В Strapi настроены **роли и права** для публичного API (Users & Permissions) так, как нужно сайту (часто отдельный API Token только на чтение для CI/сборки).
4. (Опционально) Медиа через Cloudinary — переменные `CLOUDINARY_*` на сервере Strapi.

## Переменные на стороне сайта (монорепо)

Типичные имена (зависят от вашего кода; подставьте свои):

- **`STRAPI_URL`** — базовый URL API без хвоста, например `https://admin.pantera-boxing.ru`
- **`STRAPI_TOKEN`** — токен для сборки (если контент запрашивается при `npm run build`)

Где править:

- локально: `.env` / `.env.production` в каталоге фронтенда (что у вас задано в монорепо);
- **GitHub Actions / секреты** репозитория сайта — те же переменные для production-сборки.

После смены URL пересоберите сайт и задеплойте статику.

## Безопасный порядок работ (без длинного простоя)

1. **Убедиться**, что этот репозиторий задеплоен и стабилен на VPS (см. [vps-nginx-operations.md](vps-nginx-operations.md)).
2. В репозитории **сайта** (монорепо) указать **`STRAPI_URL`** на новый домен, собрать **локально** или в CI и проверить ключевые страницы.
3. Смержить и задеплоить сайт на Beget.
4. **Только после** smoke-теста продакшена убрать из монорепо папку со старой CMS и почистить CI.

Так старый деплой сайта продолжает работать, пока вы не подтвердите, что новый API стабилен.

## Что убрать из монорепозитория

Точные пути зависят от вашей структуры; обычно:

- каталог со Strapi (раньше что-то вроде `strapi-cms/`, `admin/`, `apps/cms/` — как у вас названо);
- корневые файлы, которые только проксируют запуск Strapi из монорепо;
- джобы CI, которые **собирают или проверяют backend** вместе с фронтом — оставить только pipeline фронтенда;
- документация в монорепо: указать, что CMS вынесена в `pantera-mini-apps-admin`.

Перед удалением сделайте **ветку** и **коммит**, чтобы откатиться через `git revert` при необходимости.

## Перенос истории Git (если нужна)

Если важна история коммитов только по папке CMS, можно один раз воспользоваться `git subtree split` и запушить в `pantera-mini-apps-admin` (если история там ещё не полная). Иначе достаточно считать этот репозиторий источником истины и удалить дубль из монорепо без переноса истории.

## Rollback

Если после переключения что-то пошло не так:

1. В монорепо (или в секретах CI) временно вернуть старый **`STRAPI_URL`** / токен.
2. Пересобрать и задеплоить сайт.

Статический сайт на Beget откатывается пересборкой с прежними переменными.

## Чеклист

- [ ] Strapi на VPS отвечает по HTTPS, `PUBLIC_URL` верный
- [ ] В монорепо обновлены `STRAPI_URL` (и токен при необходимости) локально и в CI
- [ ] Сборка сайта проходит, страницы с данными из API проверены
- [ ] Продакшен-сайт задеплоен и проверен
- [ ] Папка CMS и лишние джобы удалены из монорепозитория
- [ ] Документация монорепо обновлена
